​​​​​​Overview

This addon is used to communicate with an ECU over a diagnostic protocol. Supported protocols are KWP, UDS, & GMLAN. CAN is the supported physical layer. The supported services for each protocol are listed below. Note: This addon requires a valid license for the NI Automotive Diagnostic Command Set (ADCS) 19.0+. ​

Addon Specifics

​​Supported Services

The addon reads DIDs, PIDs (GMLAN only), memory (KWP/UDS only) and DTCs. It writes DIDs, CPIDs (GMLAN only), memory (KWP/UDS only) and clears DTCs. In GMLAN, PIDs are dynamic data (values change from read to read, ie sensor data). DIDs are static data (values don't change from read to read, ie VIN number). In UDS and KWP2000, DIDs are dynamic and static data. DTCs are diagnostic trouble codes. CPIDs are output PIDs. KWP/UDS also supports IO control and routines (start, stop, status). Custom (user defiend) messages are supported by all protocols. ​More specifically, the services currently implemented are:​

UDS

  • Tester Present (x3E)
  • Diagnostic Session Control (x10)
  • Communication Control (x28)
  • ECU Reset (x11)
  • Request Seed/Send Key (x27)
  • Read Data by Identifier (x22)
  • Write Data by Identifier (x2E)
  • Read Memory by Address (x23)
  • Write Memory by Address (x3D)
  • Input Output Control By Identifier (x2F)
  • Routine Control (x31)
  • Control DTC Setting (x85)
  • Report Severity Information of DTC (x19 with x9 option)
  • Report DTC by Status Mask (x19 with x2 option)
  • Clear Diagnostic Information (x14)

KWP

  • Tester Present (x3E)
  • Start Diagnostic Session (x10)
  • Stop Diagnostic Session (x20)
  • Disable Normal Message Transmission (x28)
  • Enable Normal Message Transmission (x29)
  • ECU Reset (x11)
  • Request Seed/Send Key (x27)
  • Read Data by Local Identifier (x21)
  • Write Data by Local Identifier (x3B)
  • Read Memory by Address (x23)
  • Write Memory by Address (x3D)
  • Input Output Control By Local Identifier (x30)
  • Start Routine by Local Identifier (x31)
  • Stop Routine by Local Identifier (x32)
  • Request Routine Results by Local Identifier (x33)
  • Control DTC Setting (x85)
  • Read Status of DTC (x17)
  • Read DTC by Status (x18)
  • Clear Diagnostic Information (x14)

GMLAN Only

  • Tester Present (x3E)
  • Initiate Diagnostic Operation (x10)
  • Disable Normal Communication (x28)
  • Return to Normal Mode (x20)
  • Request Seed/Send Key (x27)
  • Read Data by Identifier (x1A)
  • Write Data by Identifier (x3B)
  • Read Data By Parameter Identifier(x22)
  • Device Control (xAE)
  • Dynamically Define Message (x2C)
  • Read Data By Packet Identifier (xAA)
  • Read Status Of DTC By DTC Number (xA9 with x80 option)
  • Read DTCs by Status (xA9 with x81 option)
  • Send On Change DTC Count (xA9 with x82 option)
  • Clear Diagnostic Information (x4)​

​​Addon Configuration

Upon adding the addon, an configuration GUI will open. Select the desired diagnostic protocol and base baud rate. Optionally, a previously exported diagnostics addon configuration can be imported.

The addon will be added to the system definition. There will sections added for all the supported services.

On the main page, you can change the CAN port, CAN mode (CAN 2.0, CAN FD, CAN FD + BRS), Baud Rate, and Termination. The search button (magnifier glass icon) will detect the available CAN ports on the current Target/Controller. 

Also, notice the built in help. This help provides the details needed to use each service.

Go to​​ the Protocol Settings tab and set the Transmit ID (master to ECU) and Receive ID (ECU to master). Diagnostic protocols are command/response based. These IDs are the specific frame command and response IDs used for the communication. They are company and ECU specific. GMLAN has a third ID for push-based messages (breaks the normal command/response paradigm). 

After these settings are entered, an initial test of the configuration is recommend. Deploy the system definition, check that the Tester Present service is working without errors, and confirm the expected Transmit and Recieve IDs are on the bus (using NI XNET Bus Monitor or the included tool's bus s​tream discussed below). If this test is not successful, make sure the ECU is powered and wired, confirm basic CAN is working, check for bus errors, confirm the CAN and protocol settings with the ECU experts, check for negative diagnostic responses, etc. If the test is successful, continue setting up the addon.

Each service has the same basic structure. There are Services Channels that control and report on the service.

​

Trigger, for example, will force the service to execute or update. Timestamp reports when it last executed. Success and Error Code report the service status.

Additionally, some service have Settings, Command, or Response sections.

 

Settings generally control or parameterize the service. Command and Response sections contain the data within the service frame.

Some services work without any additional configuration. Tester Present, for example. Other services, like reading DIDs, need more information. In most cases, there will be an Add, Import and/or Edit button.

As for importing, a basic .CSV file can be used to quickly add ma​ny items. The specific import formats are defined per service in:

C:\Users\Public\Documents\National Instruments\NI VeriStand <year>\Custom Devices\Diagnostics Addon\Windows\Import File Formats

Some of the services have special configurations on the section itself. Unlock ECU and Read DTCs are two example. For Unlock ECU, select None (ECU already unlocked), DLL (or shared object depending on the target's operating system) or use a Master Key.

​

​​In the case of Read DTCs (Start DTCs in GMLAN), set the byte length and formating of the DTCs as well as the maximum number of expected DTCs.

Additionally, import a set of numeric to string values for the DTC codes and status.

As with above, the file format examples for this are located at:

C:\Users\Public\Documents\National Instruments\NI VeriStand <Year>\Custom Devices\Diagnostics Addon\Windows\Import File Formats

Polling of particular services is supported by the addon. Polls are an application feature of the addon, not a diagnostic service. The normal operation of diagnostics is event based. However, users often want to periodically monitor a particular DID, PID, or memory location. Polls were added to make this simpler. ​​Add a poll, select the desired items to monitor, and select the rate.

 

​Similar to all other services, "Enable" the poll in Settings and "Trigger" to apply.

 

Similar to polls, there is an application feature (not a service) to do mass functions through the Command channel. As an example, if there are multiple DID inputs, normally the "Trigger" channel must be set to 1 per each DID to read the latest value. The Command channel can be set to "2" or "Read All DIDs" to do a mass reading.

 

Once everything is configured, export the configuration. This makes it so the services are defined once and can then be easily re-used in any other configurations.

​​​Addon Execution

Deploy the system definition. From within the user interface, drag and drop services as sections.

Change required channels to controls, such as Trigger and anything in Settings.

​​​

Certain channels can be added as Rings to show text. The command channel supports this.

​So do error code channels.

And so do DTC Value and DTC Status channels, if they have been setup on the Read/Start DTCs section in the system definition.
